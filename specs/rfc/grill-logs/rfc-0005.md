## The docket (draft §9.2 ODs → decisions)

### RFC5-D1 — THE KEYSTONE: enforcement stance (OD-1/OD-4/OD-5 cluster frame) — **ANDREAS CALL**
The draft's audit found: fields declared+signed but read by NO enforcement path (`frontier_ok`, `model_class`), a residency check that FAILS OPEN, and an ingress permissive branch granting a stranger MORE access than a declared partner. myelin#11's framing: without enforcement, sovereignty is "a gentleman's agreement" — the CISO objection.
**Fork:** (a) **ENFORCE** — RFC-0005 specs the sovereignty gates as normative MUSTs; every deployed gap becomes a named conformance defect fixed on the myelin#11 path; spec leads deployment behind the flag-day/window machinery. (b) **ADVISE** — v1 ratifies the fields as declarative intent only; enforcement deferred wholesale to myelin#11's own future RFC revision.
**Recommendation: ENFORCE** — security-first standing default; the ADVISE stance ratifies the gentleman's agreement as the contract, which is exactly the objection; and 0006/0007 already established spec-leads-deployment with tracked catch-up issues.

### RFC5-D2 (OD-1) — `frontier_ok` / `model_class` dead fields — follows D1
If ENFORCE: egress MUST validate model placement against the declared `frontier_ok`/`model_class`; the unsatisfiable `frontier_ok:false` + `model_class:"frontier"` combo MUST be rejected at envelope validation (a contradiction is a malformed declaration, not a runtime surprise). If the fields are judged the wrong model → retire instead (Andreas sub-call).

### RFC5-D3 (OD-2) — `max_hop`: signable field cannot be decremented → **origin-declared TTL, enforced against the observed chain**
The field stays signable + immutable (no decrement — mutation would break RFC-0004 signatures). A receiver/forwarder MUST enforce `len(signed_by chain) ≤ max_hop` — RFC-0004's chain IS the hop count observable, so no mutable counter is needed. Coherent with the ratified signing model; retires the "cannot be decremented" paradox without moving the field out of the signable set.

### RFC5-D4 (OD-3) — prefix↔classification: strict equality vs reachability budget → **DETERMINED by ratified RFC-0002 §8.3**
0002 §8.3 (Ratified, with binding vectors `prefix/aligns-local`, `prefix/mismatch-rejected`): "A subject's prefix and its envelope's `sovereignty.classification` MUST align... A mismatch is a protocol violation." **Strict equality is the normative rule; the egress engine's `CLASSIFICATION_PREFIX_BUDGET` downward-superset is the defect to fix.** The legitimate use case the budget served (internal observability copy of a public event) is served by RE-PUBLISHING as a distinct `local`-classified envelope, not by carrying one envelope onto a lower subject. 0005 cites §8.3; the draft's "specifies neither" stance is stale against the ratified sibling.

### RFC5-D5 (OD-4) — `data_residency` fail-open → **fail-CLOSED normative + closed code registry**
When `data_residency` is present, an unassigned/unknown residency code MUST reject at egress (fail-closed) — the deployed fail-open (sender evades residency gating via an unassigned code, §2.3) becomes a named conformance defect. Valid-code registry: closed set seeded from the deployed values, extended per BCP-0001. Security-first.

### RFC5-D6 (OD-5) — ingress permissive-branch trust inversion → **default ceiling for unmapped principals**
Permissive mode (`reject_unknown_partners: false`) MUST still apply a default scope/ceiling to an unmapped principal, and that default MUST NOT exceed what a declared partner's mapping could grant — declaring a partner must never REDUCE its access relative to a stranger's. The deployed unconditional-ALLOW bypass (skips §6.3 entirely) becomes a named conformance defect. Security-first.

### RFC5-D7 (OD-6) — the non-conformant nak envelope → **DETERMINED by ratified siblings**
(1) Subject: `_nak.sovereignty.*` moves under `_audit.sovereignty.*` — ratified 0002 D21, there is no top-level `_nak.` prefix. (2) `source`: MUST be an agent-class DID (ratified 0003 D16) — the enforcing stack's own identity (`did:mf:agent.{principal}.{stack}.{assistant}` of the stack running the engine); `sovereignty.engine` two-segment form is schema-invalid, gone. (3) Signed: the nak MUST carry `signed_by` — the enforcing stack signs its own verdict (0004); an unsigned enforcement verdict is forgeable (draft §10 finding). (4) The recursion exemption (nak bypasses `validateEgress`) is retained but NARROWED: the exemption applies only to `_audit.`-prefixed enforcement artifacts, stated normatively.

### RFC5-D8 (OD-7) — `sovereignty_required` matching semantics → **keep the deferral to RFC-0008 OD-5**
RFC-0008 is next in the queue and is the single normative owner of capability-matching semantics. 0005 cites forward; OD-7 closes when 0008 ratifies. (Same pattern as 0007→0010.)

### RFC5-D9 (OD-8) — `imported_principals` granularity → **principal-class DIDs** — Andreas confirm
Trust in the ingress mapping is per-PRINCIPAL: ADR-0013 (sovereign identity), RFC-0006 (per-principal admission roster), and the lookup key is `getLastStampPrincipal` (principal-granular by name). Recommend: entries are `did:mf:principal.{p}`; agent-class entries rejected at config validation. (The alternative — per-agent import entries — invites roster drift and contradicts the per-principal admission model.)

### RFC5-D10 (OD-9) — `local` = org vs principal boundary → **principal boundary (R9)**
The R9 vocabulary migration renamed operator→principal; the subject grammar's boundary segment IS the principal (`local.{principal}.>`); RFC-0006 admits per-principal. `local` means "within the publishing principal's own boundary." The envelope-schema stale text flagged by OD-9 gets the same sweep.
