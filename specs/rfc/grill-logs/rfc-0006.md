## DECIDED

### D6 — Decision-claim binding scope (OD-2 keystone) → **bind peer_pubkey + network_id (Option A)**
**Andreas, 2026-07-14.** The registry-admin's signed admit/reject claim MUST cryptographically commit to WHICH identity and WHICH network is admitted: add `peer_pubkey` AND `network_id` to the signed bytes; the registry rejects `409 identity_mismatch` if `claim.peer_pubkey != row.peer_pubkey` OR `claim.network_id != row.network_id`.

**Why A, not B:** the stress pass proved the "escalate to network_id only if global-admins-across-networks is real" trigger is ALREADY the deployed metafactory posture — a global `REGISTRY_ADMIN_PUBKEYS` admin can admit onto ANY network (`admission-requests.ts:208` short-circuit), and `network_id` is read off the same untrusted row OD-2 distrusts. The same `peer_pubkey` legitimately holds distinct PENDING rows across networks A and B (RFC §4.2 triple); binding `peer_pubkey` alone does NOT catch a request_id→wrong-network-row substitution (the check still passes — P really is the peer). Only binding `network_id` closes the cross-network confused-deputy. Threat model: a dishonest/buggy registry IS in scope → cryptographic binding beats the request_id-only + immutability-vector compensating control (a registry can violate any immutability rule it enforces itself). Security-first default.

**Conformance:** the outcome MUST be an ENFORCING vector (not a why-string) — replace `decision-claim/canonical-bytes-admit` with the widened-bytes vector + a `409 identity_mismatch` reject vector. Resolves #1748 identity coherence in the same pass.

### D7 — Migration mechanics for the binding widening → **own decoupled dual-accept window (Option A)**
**Andreas D6=A + source verification, 2026-07-14.** The binding widening runs its OWN dual-accept window, decoupled from the RFC-0001 §9 DID flag-day.

**Verification that unblocks decoupling:** the stress pass claimed binding `network_id` must fold onto the flag-day because `network_id` is a DID-encoded terminal that flips its canonical bytes at the cut. Source REFUTES this — `network_id` is a plain `string` in the registry types (`src/common/registry/types.ts:118,177`) with zero DID coupling (no `encodeDid`/`decodeDid`, no DID-class validation, and network is explicitly NOT a subject segment per CONTEXT.md). The stress agent conflated `network_id` with `principal_id` (which IS DID-encoded). Both bound fields are encoding-stable: `peer_pubkey` = raw base64 Ed25519, `network_id` = raw string. Neither flips at the flag-day → no §9 conflict → the window decouples cleanly.

**Window shape:** verifier accepts claims with OR without the bound fields, enforcing the match when present; flip to require-present after all signers upgrade. Nearly free via the #1414 verify-over-`canonicalJSON(signed.claim)`-as-received pattern (an added signed field is transparently verified by an unchanged verifier). Guard the C-style "never enforce" failure: charge the require-present flip as an explicit, dated step in the OD-2 tracking issue with a monitored narrow-claim counter that MUST read zero before the flip.

### D8 — SealedSecretWriteClaim binding scope → **widen to bind peer_pubkey (Option B)**
**Coherence with D6 (conflict #3), 2026-07-14.** The hub-admin's signed seal-delivery claim MUST bind `peer_pubkey`; the registry rejects `claim.peer_pubkey != row.peer_pubkey`. Forced by the two-party-gate coherence requirement — the gate cannot have the registry-admin claim (D6) bind identity while the hub-admin claim does not; asymmetric binding is itself a seam the completion audit flags.

**Framed precisely as defense-in-depth + audit/intent signal, NOT a fix for identity substitution** (that is D6 + PENDING-row immutability). What it buys is narrow and real: reject a claim whose bound key != row.peer_pubkey (catches a hub-admin fumbling the target row at write time) + give the member a signed intended-recipient assertion. `crypto_box_seal` still carries the PRIMARY guarantee (a blob sealed to the wrong key can't be opened). Decide consistently with RFC-0004 D14 (same "bind the vouched/target identity into the signed bytes" shape). Carry the omission only as historical v-current behind the D7 dual-accept window. Unblocks D2 Option A (per-stack seal sub-rows are unsafe without this binding).

### D2 — Seal delivery for multi-stack principals → **RFC owns per-stack seal sub-rows (Option A)**
**Andreas, 2026-07-14.** The registry sealed-secret channel OWNS per-stack delivery: §8 gains a `sealed_secrets[]` per-key shape (was a single nullable slot), each ciphertext sealed to that covered stack's key and the D8 seal claim binding its `target_stack_pubkey`. The admit stays per-principal (one decision); the SEAL becomes a per-stack write.

**Why A:** (1) hub-mode-AGNOSTIC — covers BOTH operator-mode (metafactory) and sovereign/PSK hubs (the community/halden zero-trust model), where B would leave sovereign hubs with NO channel-supported multi-stack transport; (2) it's the fix ADR-0023 Consequences already names ("the 2nd stack joins with the stack's OWN sealed .creds" = per-stack delivery) — B's principal-wide instinct is what ADR-0023 rejected (stress: the delivered code has no principal-scoped seal, and a shared one collapses per-stack subject isolation, which the v2 leaf_user guard exists to reject); (3) one uniform delivery path, not a registry-first-stack + hub-side-channel bifurcation — a second impl reproduces exactly one thing; (4) safe now — D8's peer_pubkey binding closes the wrong-target-seal fumble that made A risky in the raw docket.

**Cost (accepted):** record-shape migration (single `sealed_secret` slot → `sealed_secrets[]` keyed on target stack pubkey). Somewhat redundant with direct hub minting under operator-mode — a fair price for uniform-and-complete across both hub modes. Resolves the transport half of #1748. Coupled to D8 (decided ✓).

### D13 — Per-network admin authority home → **fold normative into RFC-0006 §7.4 (Option A)**
**Andreas, 2026-07-14.** Fold ONLY the authorization RULE into a new normative §7.4: "the registry MUST reject an admit whose signing admin is not authorized for the claim's bound `network_id` (global `REGISTRY_ADMIN_PUBKEYS` OR the network's per-network allowlist)." Retarget the phantom 'RFC-0020' refs to §7.4. Keep ADR-0020 as an INFORMATIVE pointer for the deployment MECHANISM (allowlist storage, admin-key management).

**Why A:** D6=A bound `network_id` into the decision claim's signed bytes → the authorization rule is now wire-load-bearing (a 2nd impl validating a claim MUST reproduce it; D6's enforcing vector needs a normative owner). B leaves it spec'd only in a cortex-local ADR — the reconstruct-from-code trap. C (charter a real RFC-0020) is overkill for what is currently one paragraph and would block RFC-0006 ratification on a non-existent doc. Reversible to C later via an `Updates:` RFC (BCP-0001 living-spec) if authority grows a real surface (rotation/delegation/quorum). Coherent carve with D15 (conflict #6): authority = membership-plane (normative, stays), rate-limit = substrate-plane (carves to RFC-0010).
